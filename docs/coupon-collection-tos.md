# クーポン・キャンペーン収集の規約適合性調査(issue #8)

- 調査日: **2026-07-12**(Claude Code による公開 Web の調査。法的助言ではなく事実の整理)
- 目的: クーポン収集の半自動化(issue #8)に先立ち、各社サイトの robots.txt・利用規約と情報の公開状況を確認し、収集方式の判断根拠を残す
- 収集方式の前提: Claude Code スキルが実行時(月 1 回程度・任意契機)に**少数の公開ページを閲覧・読解**して campaigns.json の下書きを作る。大量・高頻度の機械的スクレイピングは行わない。収集した事実情報(還元率・期間・対象店舗等)は GitHub 公開リポジトリの JSON として再配布される

## 結論サマリ

| 事業者 | robots.txt | 自動アクセス禁止条項 | 転載・再利用条項 | キャンペーン情報の公開状況 | 収集可否の判定 |
|---|---|---|---|---|---|
| PayPay | `/event/` 許可 | 見つからず | **再配布禁止**(第4条) | 一覧・詳細とも静的公開。大型クーポンも Web 公開 | ○ 事実情報のみなら可 |
| au PAY | ほぼ全許可(camp.auone.jp の 1 パスのみ拒否) | 見つからず | 複製・配布は許可制(著作権条項) | 一覧は JS 描画。静的な入口あり | ○ 静的入口経由で可 |
| d払い | `/keitai_payment/` 許可(`/api/` は拒否) | 見つからず | 複製・公衆送信は許可制(著作権条項) | 一覧は JS 描画(API は robots 拒否)。静的な入口あり | ○ 静的入口経由で可(API 直叩き不可) |
| 三井住友カード | キャンペーン公開ページ許可 | 見つからず(ただし WAF が非ブラウザを 403) | サイト利用規約自体が見つからず | 一覧 JSON(`cp_list.json`)が公開・構造化 | △ **半手動フロー**(内容貼り付け)で収集 |
| 三菱UFJニコス | 全許可 | 記載なし | 複製・転用・改変は禁止(表現が対象) | まとめページ+詳細ページが静的公開 | ○ 事実情報のみなら可 |
| 楽天ペイ | `/campaign/` 許可 | **クローラー・データ収集/抽出ツール・データマイニングを明示禁止** | **データ含め複製等一切禁止**(著作権非保護のデータにも権利主張) | 一覧は JS 描画で読めない | ○ **加盟店側の公表から取得**(楽天サイトへはアクセスしない) |
| Amex | /ja-jp/ 許可 | 見つからず | 複写・配信・公共目的利用を明示禁止 | 公開一覧はあるがオファー全量はログイン必須・会員ごと個別 | △ 巡回対象(全員対象と断定できる施策のみ収録) |

## 収集の共通原則(この調査から導かれるルール)

スキル実装・data/README.md の運用ルールに反映する:

1. **事実情報のみを収録する**。還元率・期間・上限・対象店舗名・出典 URL だけを自前のスキーマ・文言で JSON 化し、各社ページの説明文のコピペ・画像・バナーは一切含めない(全社の著作権/転載条項と整合させるための最重要ルール。事実データ自体は著作物ではない)
2. **フェッチ前に robots.txt を確認**し、Disallow パスはスキップする(camp.auone.jp の個別パス、docomo の `/api/` など。robots.txt は変わりうるので実行のたびに確認)
3. **アプリ内限定の情報には手を出さない**。PayPay coupon-corner ポータル、au PAY / d払いのアプリ内クーポン一覧、JS 描画一覧を裏で支える API の直叩きは行わない。ログイン不要・静的に読める公開ページのみを情報源とする
4. **低頻度・少ページ**を維持する(月 1 回程度・一覧+差分のある詳細ページ数件)
5. **403 等の拒否応答は回避しない**(UA 偽装の積み重ねはしない。サイト側の意思表示として扱い、アクセス手段を再検討する)
6. **規約・robots は変更されうる**ため、四半期程度で規約ページを再確認する
7. **加盟店サイトから収集する場合も、初出ドメインでは robots.txt とサイト規約(サイトポリシー等)を確認する**。自動アクセス禁止・特筆すべき条項があれば本ドキュメントに追記し、無ければ共通原則 1〜5 の範囲で収集する(確認済みドメインは § 加盟店サイトの確認記録 参照)
8. **自治体・公的機関ドメイン(lg.jp 等)は最も低リスクな情報源**として積極的に使う。多くの自治体サイトは政府標準利用規約(第2.0版・CC BY 4.0 互換)を採用しており出典明記での再利用が明示許可されている(municipalities.json の気象庁データと同じ整理)。robots.txt 確認+利用規約が標準系であることの確認でよく、標準系でない場合のみ加盟店同様に記録する。商工会議所・実行委員会等の民間ドメインは加盟店扱い(原則 7)

### 楽天ペイ(pay.rakuten.co.jp)— 楽天サイトへはアクセスせず、加盟店側の公表から取得

- robots.txt: 公開 `/campaign/` は許可
- 規約: 楽天グループ共通の[著作権等、当社データの保護について](https://corp.rakuten.co.jp/copyright/)が、**7 社中唯一、明示的に自動収集を禁止**している:
  > 「当社による明示的な許諾がない限り、(中略)第三者による複製等一切の利用及び、**クローラーなどのデータ収集・抽出ツールの使用、データマイニング等の行為を禁止**致します」
  さらに「価格、その他のデータを含む全てのもの」について「例え著作権法(中略)により保護されないものであっても(中略)当社が法令上の権利を有します」と、**著作権で保護されない事実データにまで権利を主張する書きぶり**。事実情報のみの JSON 化・再配布であっても他社のような整理が成立しにくい
- 技術面でも一覧は JS 描画で静的取得不可(サービス別に一覧が分散)
- 論点は 2 つに分けて整理した:
  1. **取得方法**: URL を指定して Claude Code(ツール)が単発 fetch する形でも「クローラー**などの**データ収集・抽出ツールの使用」という広い文言に該当し得る読みが残る。**ユーザー自身がブラウザで閲覧し、ページ内容(テキスト/スクショ)を貼る形なら、楽天のサーバーにアクセスするのは人間+通常ブラウザのみ**となり、この条項に触れる余地がなくなる
  2. **事実データの再配布**: 取得方法に関わらず、還元率・期間等の事実を GitHub の JSON として公開する行為は「データ含め複製等一切禁止」の文言と緊張関係が残る。ただし著作権法上、事実データそのものは保護されず(楽天の主張は規約=契約ベースで、単なる閲覧者をどこまで拘束できるかは疑わしい類のもの)、収録は自前スキーマ・自前文言の事実の記録に限る(共通原則 1)。ゼロリスクではないが実務的リスクは小さい部類、と整理
- **リスクをさらに下げる経路 — 加盟店側の公表から取得する**: 楽天ペイの期間限定施策は多くが加盟店との共同施策で、**加盟店自身が自社サイトで事実を公表している**(実例: 松屋創業60周年 15%還元は [松屋フーズの告知ページ](https://www.matsuyafoods.co.jp/matsuya/whatsnew/campaign/160563.html) に還元率・期間・最低決済額・付与上限・要エントリー・対象ブランド・PA 店舗除外まで全て掲載されていることを 2026-07-12 に確認済み)。楽天の規約が対象とするのは「**楽天グループのサービスを通じて提供される**コンテンツ」なので、**加盟店が自社サイトで公表した事実を採る限り、取得(論点 1)・再配布(論点 2)とも楽天規約の射程外**になる。加盟店サイト自体の著作権条項には共通原則 1(事実情報のみ・出典 URL 記録)で対応する
- **判断(2026-07-12 決定): スキルは楽天ペイのサイト(pay.rakuten.co.jp / checkout.rakuten.co.jp 等の楽天グループドメイン)に一切アクセスしない**。運用フローは次のとおり:
  1. **発見はユーザーのヒント**: ユーザー自身が通常のブラウザで楽天のキャンペーン一覧を確認し、「今、くら寿司とサンマルクカフェでキャンペーンがあるはず」のようなヒントをスキルに与える(ヒントが無ければ楽天ペイ分は収集しない。Web 検索・加盟店ニュースページでの発見も併用可)。楽天サイトへのアクセスは人間+通常ブラウザのみなのでツール使用条項に触れず、ヒント自体は事実の伝達であって楽天コンテンツの複製でもない
  2. **事実の取得は加盟店側の告知ページ**(主経路)。スキルがヒントを手がかりに加盟店サイトを検索・読解し、`detail_url` も加盟店側 URL を使う
  3. 加盟店側に載っていない細部(付与時期・対象決済手段の内訳等)がどうしても必要な場合のみ、**内容貼り付け方式**(ユーザーがブラウザで楽天ページを見て内容を貼る → スキルは構造化のみ)で補完する。補完しない場合は当該項目を収録しない/要確認扱いにする
  4. **自治体施策は自治体側情報源から収録できる**: 他決済手段の入口で見つけた自治体キャンペーンについて、自治体公式ページ(共通原則 8)で対象決済手段を確認し、楽天ペイが含まれていれば楽天ペイぶんのエントリも作る(自治体クロスチェック)。事実の出所が自治体の公表なので楽天規約の射程外

### American Express(americanexpress.com)— 「限定的に可」・手動キュレーション向き

- robots.txt: `/ja-jp/` 系キャンペーンページは許可
- 規約: [ウェブサイト規約](https://www.americanexpress.com/jp/company/legal/web-rules.html)に自動アクセスの名指し禁止は見つからず。ただし「掲載情報全部または一部に関し(中略)複写、配信(中略)商業的・公共的な目的のための利用は、禁止」→ 共通原則 1 で対応しつつ、条項の書きぶりが広い点に留意
- 公開状況(フィージビリティ評価: **限定的にある**):
  - ログイン不要の[カード会員様限定キャンペーン一覧](https://www.americanexpress.com/ja-jp/benefits/lifestyle/event-campaign/campaign/)は存在(十数件規模。店名・還元額・期間・上限は取れる)
  - **アメックス・オファーは会員ごとにパーソナライズ**(公式に「カード会員様により対象のキャンペーンが異なります」)で、全量・対象判定はログイン必須。公開ページに出るのは「一例」のみ
  - 全員対象の大型ブランドキャンペーン(SHOP SMALL 型)は実施されれば公開告知されるが、**直近 1〜2 年はキャッシュバック型が休止気味**(全国規模は 2023 年が直近)。発生頻度は年数回以下
  - 加盟店サイト巡回は SN 比が悪く体系的経路にならない。ポイ活ブログは情報源として存在するが転載・正確性の問題があり一次情報にしない
- **収録上の注意**: Amex 施策の多くは「要エントリー(ログイン)」「対象は会員により異なる」ため、収録するなら免責注記が必須(現行スキーマではまず `conditions` の文章で表現し、件数が増えるようなら `entry_required` フラグの追加を検討)
- **判断(2026-07-12 決定): 公開一覧を巡回対象に含める**。JSON 化するのは全員対象と断定できる施策(SHOP SMALL 型)のみとし、会員限定・要ログイン確認のものは見送り一覧に回す

- 結論: **加盟店側取得の対象として問題なし**

## 決定事項(2026-07-12・ユーザー判断)

- **楽天ペイ**: スキルは楽天グループのサイトにアクセスしない。**発見はユーザーが楽天一覧を人力確認して与えるヒント**(「今○○と△△でキャンペーンがあるはず」。無ければ収集しない)、**事実の取得は加盟店側の告知ページ**(共同施策のため加盟店が自社サイトで公表している。楽天規約の射程外)。楽天ページにしか無い細部の補完のみ「内容貼り付け方式」(ユーザーがブラウザで見て貼る→スキルは構造化のみ)で行う。**自治体施策は自治体クロスチェック**(自治体公式ページで対象決済手段を確認)により楽天ペイぶんも収録できる。既収録の松屋 60 周年施策は事実・`detail_url` とも松屋フーズの告知ページ由来であることを確認済みで、この方式に既に適合している(松屋側の規約確認も済み。§ 加盟店サイトの確認記録)
- **Amex**: 公開キャンペーン一覧を巡回対象に含める。全員対象と断定できる施策のみ JSON 化し、会員限定・要ログイン確認のものは見送り一覧へ
- **SMCC**: WAF の 403 は迂回せず、半手動フロー(ユーザーが `cp_list.json`・詳細ページをブラウザで開いて貼る → スキルが JSON 化)で対応
