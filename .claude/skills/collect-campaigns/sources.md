# 情報源カタログ

collect-campaigns スキルが巡回する入口と、ソース別のアクセス方式・読み方。規約上の判断根拠は docs/coupon-collection-tos.md(正本)を参照。入口 URL が死んでいたら Web 検索で後継ページを探し、見つけたらこのファイルの更新も下書きに含める。

## PayPay — 直接閲覧可

| 入口 | URL |
|---|---|
| キャンペーン一覧 | https://paypay.ne.jp/event/ |
| 自治体キャンペーン一覧 | https://paypay.ne.jp/event/support-local |

- 一覧は「開催中/開催予定/開催終了」区分の静的ページ。詳細は `/event/<slug>/`(slug は `都道府県-市名-日付` 形式が多く、id 命名の参考になる)
- 自治体一覧は地域ブロック→都道府県別。期間・種別(ポイント還元/商品券)は一覧で取れ、率・上限は詳細ページで補完する。**商品券型は収録対象外**(ポイント還元型のみ)
- 大型の全員配布クーポン(スーパーPayPayクーポン・ジャンボ等)は `/event/` 配下に公開ページがある。**アプリ内の個別店舗クーポン(coupon-corner)は対象外**
- サイト内検索(`/search/`、`?s=`)は robots.txt で Disallow。使わない

## 三井住友カード — 半手動(ツールアクセス禁止)

- WAF が非ブラウザアクセスを 403 で弾く。**迂回せず**(ブラウザ自動操作も含めて機械アクセスはしない)、ユーザーに「ページをファイルに落とす」ことだけ依頼する。**ユーザーにページを読ませたり要約させたりしない**(読解はスキルの仕事)
- 落とし方は **HTML 保存(ページ全体のファイル保存)に統一**する。テキスト貼り付け(`Cmd+A`→`Cmd+C`)は**リンク先 URL(href)と表組みの中身が落ちて全情報を収集できない**ことが実運用で確認済みのため、依頼文では使わない(ユーザーがテキスト貼り付けで渡してきた場合のみ受け取り、照合できなかった項目(リンク URL・表)を報告に明記する)
  - **依頼する手順**: ブラウザでページを開く → `Cmd+S` → 形式「ウェブページ、HTML のみ」で保存 → 保存先パスを教えてもらう(スキルがそのファイルを読む)。cp_list.json のような JSON URL も同様に `Cmd+S` でそのままファイル保存できる
- 新規収集の流れ:
  1. ブラウザで https://www.smbc-card.com/memfs/campaign/json/cp_list.json を開き、`Cmd+S` でファイル保存してパスを教えてもらう(タイトル・期間・要エントリー・詳細 URL が構造化された公開 JSON。約 80 件)
  2. スキルがファイルを読み、候補(既存会員向け・還元系・収録基準を満たしそうなもの)を絞り込み、**必要な詳細ページ(`/memfs/campaign/cp_XXXX.jsp`)を具体的に指名**する(通常 2〜3 ページ)
  3. ユーザーが HTML 保存で落とし、スキルが読解・JSON 化する
- メンテ実行時の既存施策(`smcc_combini_restaurant` 等)は、`detail_url` のページを **HTML 保存**で落としてもらい、率・条件・対象チェーンに加えて **`store_list_url` などの収録済みリンク URL・表の中身(付与時期等)まで**照合する
- **リンク URL の変更を検知しても即座に上書きしない**。収録 URL の目的は「ユーザーがその情報(対象店舗一覧等)を実際に見られること」なので、新旧両方の URL をユーザーにブラウザで開いてもらい、**中身が目的に合う方**を採用する(公式ページ側の誤リンクがあり得る)
  - 既知の誤リンク(2026-07-12 確認): 公式キャンペーンページの「ケンタッキーフライドチキンの対象店舗一覧はこちら」は付与条件ページ(`/mem/platinum-preferred/special-store/kfc.jsp`)に誤リンクしている。実際の店舗一覧は収録済みの `/mem/wp/store_list/kfc.jsp`(こちらを維持)。公式側が修正されたらこの注記を消す
- 入会キャンペーン(新規入会特典)は収録対象外(既存会員の店舗還元のみ)

## Amex — 直接閲覧可(全員対象と断定できるもののみ)

| 入口 | URL |
|---|---|
| キャンペーン一覧 | https://www.americanexpress.com/ja-jp/benefits/lifestyle/event-campaign/campaign/ |

- JSON 化するのは**イシュアー不問・全会員対象と公開ページから断定できる施策**(SHOP SMALL 型)のみ。会員限定・要ログイン確認・「対象は会員により異なる」ものは見送り一覧へ
- 帰属は `card_brand: "Amex"`(card_id / payment_method_id は null)
- 要エントリーの場合は `conditions` に「事前エントリー必要(Amex 会員サイト)」を明記する
- 大型施策は年数回以下。収穫ゼロが通常なので、無ければ「Amex: 該当なし」と報告するだけでよい

## 楽天ペイ — 楽天ドメインへのアクセス禁止。ヒント方式+加盟店/自治体側取得

- **発見**: ユーザーの `rakuten:` ヒント(「今くら寿司とサンマルクカフェでやってるはず」等)が起点。ヒントが無ければ楽天ペイの promotion は収集しない(自治体クロスチェックで判明した municipal 分は除く)
- **取得**: ヒントのチェーン名で Web 検索し、**加盟店側の告知ページ**(例: 松屋フーズのニュースページ)から事実を取る。`detail_url` も加盟店側 URL を使う
- 初出の加盟店ドメインはガードレール 6(robots.txt+規約確認と記録)を実施
- 加盟店側に無い細部(付与時期・対象決済手段の内訳等)が必要なときだけ、ユーザーに楽天ページの内容貼り付けを依頼する。貼り付けが無ければその項目は収録しない/要確認扱い

## AEON Pay — イオン系ドメインへのツールアクセス禁止。自治体クロスチェック+PR TIMES/加盟店側取得(必要時のみ半手動)

- **イオン系ドメイン(aeon.co.jp / aeon.com 等)にはツールでアクセスしない**(「指定した方法以外の利用」の一般条項+WAF の 403 を意思表示として尊重。docs/coupon-collection-tos.md 参照)。SMCC と同じ運用
- **主な収録経路は自治体クロスチェック**(自治体還元事業の定番決済になっており、自治体公式ページの対象決済リストに AEON Pay があれば `aeon_pay` のエントリを作る。lg.jp 側で完結するのでイオン系ドメインへのアクセスは不要)
- 単独施策の**発見**は PR TIMES・加盟店側告知・ユーザーヒントから(Web 検索: 「AEON Pay 還元 キャンペーン site:prtimes.jp」等)。ただし単独施策は**抽選型・新規登録特典・イオングループ限定が中心**で、収録基準(全員対象・定率還元)を満たすものは少ない。収穫ゼロが通常なので、無ければ「AEON Pay: 該当なし」と報告するだけでよい
- イオン側ページ(iAEON キャンペーン一覧 https://www.aeon.com/aeonapp/campaign/ や個別 LP)にしか無い事実が必要な場合は、**半手動フロー(HTML 保存。手順は SMCC セクション)** でユーザーに依頼する。`detail_url` にイオン系 URL を収録するのは可(ユーザーがブラウザで見る分には問題ないため)
- 還元先は AEON Pay 残高が多い(ポイントでなく残高付与)。条件欄に書き分ける

## メルペイ — メルカリグループドメインへのアクセス禁止。PR TIMES+加盟店/自治体側取得

- **メルカリグループのドメイン(mercari.com / jp.mercari.com / jp.merpay.com / help.jp.mercari.com 等)にはアクセスしない**(ガイドの禁止行為「インターフェイスとは別の手法でのアクセス」に該当し得るため。docs/coupon-collection-tos.md 参照)。`detail_url` にもメルカリ側 URL は使わない
- **発見**(楽天と違いユーザーヒント不要で自走できる):
  - PR TIMES のメルカリ社リリース(Web 検索: 「メルペイ 還元 キャンペーン site:prtimes.jp」等)
  - 加盟店側の告知(サンドラッグ・すかいらーくグループ等はメルペイ大型還元の常連)
  - ユーザーヒントも従来どおり受け付ける
- **取得**: PR TIMES リリース・加盟店側告知ページ・自治体公式ページから事実を取る。初出の加盟店ドメインはガードレール 6(robots.txt+規約確認と記録)を実施
- メルカリ側ページにしか無い細部が必要なときだけ、ユーザーに内容貼り付けを依頼する(貼り付けが無ければその項目は収録しない/要確認扱い)
- 定期開催の「主要チェーン最大 50% 還元」は**加盟店ごとに付与上限が別**(例: 2026-06 は各加盟店につき上限 1,000 ポイント)なのが定型。上限の掛かり方(加盟店ごと/合算)をリリースで必ず確認する。**メルカード・iD・バーチャルカード決済は対象外**が定型条件なので、対象決済方法の内訳も確認する
- 還元はメルカリポイント(有効期限あり。60〜180 日が多い)。条件欄に書き分ける

## 自治体公式ページ(クロスチェック用)— 直接閲覧可

- 発見: Web 検索「{自治体名} キャッシュレス キャンペーン {年}」「{自治体名} {決済名} 還元」等。lg.jp・自治体公式ドメインを優先する
- 多くの自治体サイトは政府標準利用規約(第2.0版・CC BY 4.0 互換)採用で、出典明記での再利用が明示許可されている(ガードレール 7)
- 対象決済手段の全リスト・中小/大手の率区分(→ `rate_rules`)・上限は自治体ページが一次情報。決済会社ページと食い違う場合は自治体ページを優先し、報告に食い違いを明記する
- **特設サイト**(実行委員会運営の `{名称}-cashless.jp` 等の民間ドメイン)にしか無い事実(率区分・対象外リスト等)が必要な場合は、ガードレール 6 の手順で robots.txt+規約を確認し、問題なければそのまま収集する(人間への確認は不要)。問題あり・判断不能のときだけ要確認としてレビューに回す
