# collect-campaigns — 施策データの収集・メンテナンス

還元施策データ(data/campaigns.json)の新規収集と既存メンテを行い、**人間レビュー可能な下書き(working tree への編集+サマリー報告)まで**を作る。コミット・プッシュは絶対にしない(ユーザーのレビューと指示を待つ)。

必読の前提ドキュメント(実行のたびに参照):

- `docs/coupon-collection-tos.md` — 規約適合性の判断根拠と収集原則。**本スキルのガードレールの正本**
- `data/README.md` — スキーマ仕様・収録基準・運用ルール
- `.claude/skills/collect-campaigns/sources.md` — 情報源カタログ(入口 URL・アクセス方式・ソース別の読み方)
- `.claude/skills/collect-campaigns/mapping.md` — 事実 → campaigns.json への変換規則

## 引数

```
/collect-campaigns                          # フル実行(全ソースの新規収集+メンテ)
/collect-campaigns paypay|aupay|dpay|card|amex|municipal   # ソース限定
/collect-campaigns maintenance              # 既存データのメンテのみ
/collect-campaigns municipal 関東           # 自治体を地域ブロック限定で(北海道・東北/関東/中部/近畿/中国・四国/九州・沖縄)
/collect-campaigns rakuten: くら寿司 サンマルクカフェ   # 楽天ヒント(このヒントがある時だけ楽天ペイを収集)
```

- `card` = 三井住友カード(半手動)+三菱UFJニコス
- 楽天ペイは **`rakuten:` ヒントが与えられた場合のみ**対象(自治体クロスチェックでの検出は除く。§ 自治体クロスチェック)
- 複数指定可(例: `/collect-campaigns paypay municipal 関東`)

## ガードレール(違反禁止。docs/coupon-collection-tos.md の共通原則を実行ルール化したもの)

1. **事実情報のみ収録する**。還元率・期間・上限・条件・対象店舗名・出典 URL を自前の文言で構造化する。各社ページの説明文のコピペ・画像の取り込みは禁止
2. **楽天グループのドメインにアクセスしない**(pay.rakuten.co.jp / checkout.rakuten.co.jp / corp.rakuten.co.jp 等すべて)。楽天ペイ施策の事実は加盟店・自治体側の公表からのみ取得する
3. **三井住友カード(smbc-card.com)にツールでアクセスしない**。403 を UA 偽装等で迂回しない。半手動フロー(sources.md 参照)を使う
4. **アプリ内限定情報・裏 API に手を出さない**。PayPay coupon-corner、au PAY / d払いのアプリ内クーポン、JS 描画一覧を支える API(docomo `/api/` 等)の直叩きは禁止
5. **フェッチ前に対象ドメインの robots.txt を確認**し(1 実行内でドメインごとに 1 回でよい)、Disallow パスはスキップする
6. **初出の民間ドメイン**(加盟店・キャンペーン実行委員会・特設サイト等)は robots.txt+サイト規約を確認し、結果で分岐する:
   - **問題なし**(自動アクセス禁止条項が無い) → 人間に確認を求めず**そのまま収集を続行**し、確認結果を docs/coupon-collection-tos.md「加盟店サイトの確認記録」への追記として下書きに含める
   - **問題あり・判断不能**(禁止条項がある、規約が見つからず判断に迷う、楽天グループ等の禁止ドメイン) → そのドメインからは収集せず、取れなかった事実を「要確認」としてレビューに回す(ユーザーの目視確認か内容貼り付けを依頼する)
7. **自治体・公的機関ドメイン**(lg.jp 等)は robots.txt 確認+利用規約が政府標準利用規約(CC BY 4.0 互換)系であることの確認でよい(個別記録は不要。標準系でない規約なら加盟店同様に記録)。商工会議所・実行委員会等の民間ドメインは加盟店扱い(規則 6)
8. **低頻度・少ページ**を守る。一覧+差分のある詳細ページのみ読む。同一ページの再取得を繰り返さない

## 実行フロー

### Phase 0 — 現状把握

1. `data/campaigns.json`・`data/merchants.json`・`data/payment_methods.json` を読む
2. 今日の日付で各施策の状態を整理: ACTIVE / UPCOMING / 終了(30 日以内) / 終了(30 日超=削除候補)
3. 引数を解釈して対象ソース・地域ブロックを確定する

### Phase 1 — 既存メンテ(`maintenance` またはフル実行時)

各収録済み施策について:

1. `detail_url` をソース別アクセス方式(sources.md)に従って再確認する。SMCC 由来・楽天ドメインの URL はユーザーに内容貼り付けを依頼する
2. **改定検知**: 率・期間・上限・条件・**対象チャネル**(店頭/アプリ注文限定の別。mapping.md「対象チャネルの確定」)が公表と食い違えば修正下書きを作る
3. **早期終了検知**: 予算到達・早期終了の告知があれば `period_end` 修正または削除提案
4. 確認できた施策の `verified_date` を実行日に更新する(確認できなかったものは据え置き、理由を報告)
5. **終了後 30 日超**の施策は削除の下書きを作る(data/README.md「期限切れデータ」ルール)

### Phase 2 — 新規収集

sources.md のソースごとの手順で入口を巡回し、未収録の施策候補を集める。候補ごとに記録するもの: 施策名(自前要約)・事実(率/額・期間・上限・条件・対象チェーン/自治体)・出典 URL・取得日。

自治体施策は必ず § 自治体クロスチェック を通す。

### Phase 3 — フィルタと下書き

1. **収録基準**(data/README.md): 還元率 5% 以上(自治体・カード会社期間限定)。クーポンは割引率 5% 以上 OR 割引額 100 円以上 AND 全員配布 AND 主要チェーン対象
2. 基準内 → mapping.md の変換規則で `data/campaigns.json` に追記する(working tree に直接編集。git diff がレビュー画面になる)。`updated_at` も更新する
3. 対象チェーンが merchants.json に無い場合は merchant エントリの下書きも作る(`reading`・`aliases` 必須。`category`・`yolp_search` は判断根拠が無ければ要確認マークを付けて報告)
4. 基準外・収録不能 → **見送り一覧**へ(理由付き: 基準未満/ネット限定/会員限定・要ログイン/アプリ内限定/断定不能 等)。ネット限定施策(d曜日等)は現時点ではスキーマに区別が無いため一律見送り
5. 抽選型は `lottery`(rate_base/discount_amount とも null)、定例日型は `recurrence`、自治体は原則 `may_end_early: true`、複数率は `rate_rules`+最大値を `rate_base`、など mapping.md の規則に厳密に従う

### Phase 4 — 検証と報告

1. `./gradlew :app:testDebugUnitTest` を実行し整合性テスト(参照切れ・フィールド排他・rate_base=rate_rules 最大値)を通す。失敗したら下書きを直して再実行
2. サマリー報告を出す(§ 報告フォーマット)
3. **コミットしない**。レビュー後の指示を待つ

## 自治体クロスチェック

自治体施策を 1 件見つけたら(どの決済手段の入口経由でも):

1. 自治体公式ページを探す(Web 検索: 「{自治体名} キャッシュレス キャンペーン {年}」等。lg.jp / 自治体公式ドメインを優先)
2. 公式ページで **対象決済手段の全リスト** と率・期間・上限を確認する(一次情報として決済会社ページより優先)
3. 対象決済手段のうち payment_methods.json の `qr_payments` にあるもの**それぞれ**についてエントリを作る(例: 湯沢市 = `yuzawa_paypay_2026_07` + `yuzawa_aupay_2026_07`)。**楽天ペイもここで収録できる**(自治体側情報源なので楽天規約の射程外。ガードレール 2 は維持=楽天ドメインには行かない)
4. `detail_url` は決済手段ごとの案内ページがあればそれ、無ければ自治体公式ページを使う
5. 自治体公式ページが見つからない・対象決済手段が読み取れない場合は、確認できた決済手段のみ収録し、他は断定せず「要確認」として報告する(不確かな情報で好条件を提示しない方針)
